# Clinical Data Reconciliation Engine

A full-stack clinical data tool that uses AI to reconcile medication records from multiple sources and validate patient data quality.

---

## How to Run Locally

### Prerequisites

- Node.js 20.x
- npm

### Backend (NestJS)

```bash
cd backend
npm install
```

Create a `.env` file in the `backend` directory:

```env
GEMINI_API_KEY=your_gemini_api_key
FRONTEND_URL=http://localhost:3000
API_KEY=your_api_key
```

Start the development server:

```bash
npm run start:dev
```

The API will be available at `http://localhost:8000/api`.

### Frontend (Next.js)

From the root directory:

```bash
cd frontend
npm install
```

Create a `.env.local` file in the `frontend` directory:

```env
NEXT_PUBLIC_API_URL=http://localhost:8000/api
NEXT_PUBLIC_API_KEY=your_api_key
```

Start the development server:

```bash
npm run dev
```

The app will be available at `http://localhost:3000`.

## LLM API Used

**Google Gemini** (`@google/genai`)

Gemini was chosen for several reasons:

- **Generous free tier** — Gemini's free tier is well-suited for a project of this scope, making it practical for development and demonstration without incurring costs.
- **Strong reasoning on structured data** — Gemini performs well on tasks that require multi-source comparison and structured JSON output, which is central to medication reconciliation.
- **Native JSON mode** — Gemini supports constrained JSON output, making it straightforward to return typed, parseable responses without prompt engineering workarounds.
- **Speed** — Gemini Flash in particular offers low latency, which improves the UX for a clinical tool where responsiveness matters.

## Key Design Decisions and Trade-offs

### Monorepo Structure

The frontend and backend live in the same repository. This simplifies development and deployment coordination but means the repo is larger and build configuration is more complex, as seen with the Render deployment setup.

### NestJS for the Backend

NestJS was chosen for its modular architecture, built-in support for dependency injection, and strong TypeScript support. It adds some boilerplate compared to a plain Express server, but the structure scales well and makes testing straightforward.

### Zod for Form Validation

Zod schemas are defined flat to match the form's field structure, with transformation to the API's expected shape happening in `onSubmit`. This keeps the form simple and avoids fighting TypeScript's type system around input vs output types.

### Caching Layer

A simple in-memory cache using `cache-manager` is used to avoid redundant LLM calls for identical requests. The cache key is generated by hashing the serialized request payload using `fast-json-stable-stringify` to ensure consistent key generation regardless of property ordering. The trade-off is that the cache is not persistent across restarts — a Redis layer would be needed for production.

### Source Reliability Weighting

The reconciliation logic weights medication sources by their stated reliability (`high`, `medium`, `low`) and recency. This is a deliberate simplification — in a real clinical system, source weighting would be governed by institutional policy and more granular metadata.

### No Database

There is no persistent storage in this version. All state is ephemeral. This keeps the architecture simple but means reconciliation results and audit logs are not stored. Adding a database would be the first step toward a production-ready system. Note a database is completely unnecessary for this version of the product as we are not performing CRUD operations.

## What I'd Improve With More Time

- **Persistent storage** — Store reconciliation results and validation reports in a database (PostgreSQL) for audit trail and historical review.
- **Authentication** — Add proper user authentication (JWT or OAuth) rather than a simple API key header.
- **Redis caching** — Replace the in-memory cache with Redis for persistence across deployments and horizontal scaling.
- **Richer LLM prompting** — Improve the system prompts with few-shot examples and more clinical context to increase accuracy and confidence scores.
- **Streaming responses** — Stream the LLM response to the frontend for a better UX on slower connections.
- **Accept/Reject workflow** — Wire up the accept/reject buttons in the result dialogs to actually persist decisions and trigger downstream actions.
- **Test coverage** — Expand unit and integration tests, particularly around the LLM service and cache key generation edge cases.
- **Error handling** — More granular error responses from the API with clinical-context-aware messages rather than generic 500s.
